# Практична робота: процеси, пріоритети та моніторинг ресурсів

## Завдання 1. Огляд активних процесів

| # | Дія | Команда |
|---|------|---------|
| 1 | Вивести список **усіх** процесів | `ps aux` |
| 2 | Інтерактивний монітор | `top` (або `htop`) |
| 3 | Знайти процес, що з'їдає найбільше RAM | `ps aux --sort=-%mem \| head` (у `top` — клавіша `M`) |
| 4 | PID поточної оболонки | `echo $$` |

**Приклад: ТОП процесів за пам'яттю (`ps aux --sort=-%mem | head`):**

```
USER    PID   %CPU %MEM    VSZ     RSS    STAT COMMAND
murad   51813  1.6 11.5 5031660 3769532  Sl   node .../tsserver.js   ← найбільше RAM
murad   43595  4.8 10.7 68249748 3488612 Sl+  next-server (v15.5.13)
murad   25738 80.0 10.0 7527448 3267624  SLl  datagrip
murad    6485  7.0  9.6 13867624 3153476 Sl+  qemu-system-x86_64
```

> Найпрожерливіший процес — **`node` (tsserver)**: `%MEM = 11.5`, тобто `RSS ≈ 3.6 ГБ`
> фізичної пам'яті. У `top`/`htop` той самий результат дає сортування за пам'яттю
> (у `top` натиснути **`M`**, у `htop` — `F6 → MEM%`).

**PID поточної оболонки (`echo $$`):**

```
$ echo $$
120043
```

**Пояснення:**

- **`ps aux`** показує **всі** процеси всіх користувачів (`a` — чужі теж, `u` —
  зручний формат із `USER/%CPU/%MEM`, `x` — навіть без термінала). Колонка `RSS` —
  реально зайнята фізична пам'ять у КБ, `%MEM` — її частка від усієї RAM.
- **`top`** — живий монітор у реальному часі (оновлюється щосекунди). Корисні клавіші:
  `M` — сортувати за пам'яттю, `P` — за CPU, `k` — вбити процес, `q` — вихід.
  `htop` — зручніша кольорова версія з мишкою та прокруткою.
- **`$$`** — спеціальна змінна bash, що містить **PID самої оболонки**. Отже
  `echo $$` друкує PID вашого bash/zsh. Перевірити: `ps -p $$`.

## Завдання 2. Робота у фоні та керування процесами

| # | Дія | Команда |
|---|------|---------|
| 1 | Запустити довгу команду у фоні | `sleep 1000 &` |
| 2 | Переглянути фонові завдання | `jobs -l` |
| 3 | Повернути процес на передній план | `fg %1` |
| 4 | Зупинити (призупинити) процес | `Ctrl+Z` |
| 5 | «Примусово» завершити | `kill %1` (або `kill -9 <PID>`) |
| 6 | Стійкий до закриття термінала запуск | `nohup <команда> &` |

**Приклад сесії:**

```
$ sleep 1000 &
[1] 120064                       # [номер_завдання] PID

$ jobs -l
[1]+ 120064 Running    sleep 1000 &

$ fg %1                          # повернули на передній план
sleep 1000
^Z                              # Ctrl+Z — призупинили
[1]+ Stopped    sleep 1000

$ kill %1                        # завершили завдання №1
[1]+ Terminated  sleep 1000

$ nohup sleep 500 &              # переживе закриття термінала
[1] 120069
nohup: ignoring input and appending output to 'nohup.out'
```

**Пояснення:**

- **`&`** наприкінці команди запускає її **у фоні** — термінал одразу вільний.
  Система друкує `[1] 120064`: `[1]` — номер **job** в межах оболонки, `120064` — **PID**.
- **`jobs`** показує фонові/зупинені завдання **поточної оболонки** (`-l` додає PID).
  Звертатись до них можна через `%1`, `%2` тощо.
- **`fg %1`** (foreground) повертає завдання на передній план; **`bg %1`** —
  навпаки, відновлює зупинений процес у фоні.
- **`Ctrl+Z`** надсилає сигнал `SIGTSTP` — **призупиняє** (не вбиває) процес,
  він стає `Stopped`.
- **`kill`** надсилає сигнал. За замовчуванням `SIGTERM` (15) — «ввічливе» прохання
  завершитись. Якщо процес не реагує — `kill -9 <PID>` (`SIGKILL`) завершує
  **примусово**, без можливості прибрати за собою.
- **`nohup`** (no hangup) ігнорує сигнал `SIGHUP`, який надсилається процесам при
  закритті термінала. Тому `nohup команда &` **продовжує працювати** після виходу;
  весь вивід пишеться у файл `nohup.out`.

## Завдання 3. Пріоритети та обмеження

| # | Дія | Команда |
|---|------|---------|
| 1 | Запуск зі зниженим пріоритетом (більший `nice`) | `nice -n 10 <команда>` |
| 2 | Змінити пріоритет уже запущеного процесу | `renice -n 15 -p <PID>` |
| 3 | Переглянути обмеження ресурсів користувача | `ulimit -a` |

**Приклад: nice + renice:**

```
$ nice -n 10 sleep 200 &
[1] 120074
$ ps -o pid,ni,cmd -p 120074
    PID  NI CMD
 120074  10 sleep 200            # nice value = 10

$ renice -n 15 -p 120074
120074 (process ID) old priority 10, new priority 15
$ ps -o pid,ni,cmd -p 120074
    PID  NI CMD
 120074  15 sleep 200            # пріоритет змінено на 15
```

**Приклад: `ulimit -a` (обмеження ресурсів):**

```
core file size          (blocks, -c) 0
data seg size           (kbytes, -d) unlimited
scheduling priority             (-e) 0
file size               (blocks, -f) unlimited
pending signals                 (-i) 126506
max locked memory       (kbytes, -l) 4069892
open files                      (-n) 1048576
stack size              (kbytes, -s) 8192
cpu time               (seconds, -t) unlimited
max user processes              (-u) 126506
virtual memory          (kbytes, -v) unlimited
```

**Пояснення:**

- **`nice`** задає «ввічливість» процесу до інших. Діапазон `-20…+19`:
  **більший** `nice` = **нижчий** пріоритет (процес поступається CPU іншим);
  **менший/від'ємний** = вищий пріоритет. `nice -n 10 команда` стартує з `NI=10`.
- Звичайний користувач може лише **знижувати** пріоритет (збільшувати `nice`).
  Для від'ємних значень (вищий пріоритет) потрібен `sudo`.
- **`renice`** змінює пріоритет **вже запущеного** процесу за його PID:
  `renice -n 15 -p <PID>`. Можна цілою групою: `-g <GID>` або користувача `-u <user>`.
- **`ulimit`** показує/задає ліміти ресурсів на процеси користувача: `-n` — макс.
  відкритих файлів, `-u` — макс. процесів, `-s` — розмір стека, `-f` — макс. розмір
  файлу, `-c` — розмір core-dump (`0` — вимкнено). `ulimit -a` виводить усі одразу.
  М'які ліміти — `ulimit -Sa`, жорсткі — `ulimit -Ha`.

## Завдання 4. Моніторинг ресурсів

| # | Дія | Команда |
|---|------|---------|
| 1 | Використання дискового простору | `df -h` |
| 2 | Вільна/зайнята RAM у зручному форматі | `free -h` |

**Дисковий простір (`df -h`):**

```
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda3       449G  239G  187G  57% /              ← коренева ФС: 57% зайнято
tmpfs            16G  311M   16G   2% /dev/shm
/dev/sda4       1.1G  6.2M  1.1G   1% /boot/efi
tmpfs           3.2G  9.7M  3.1G   1% /run/user/1000
```

**Оперативна пам'ять (`free -h`):**

```
               total        used        free      shared  buff/cache   available
Mem:            31Gi        20Gi       6.3Gi       5.4Gi        10Gi        10Gi
Swap:          7.6Gi       2.1Gi       5.6Gi
```

**Пояснення:**

- **`df -h`** (disk free) показує зайняте/вільне місце по **файлових системах**.
  `-h` (human-readable) дає `G/M` замість блоків. Колонка `Use%` — відсоток
  заповнення; тут коренева ФС `/dev/sda3` зайнята на **57%** (239 ГБ із 449 ГБ).
  Розмір окремого каталогу рахує інша команда — `du -sh <каталог>`.
- **`free -h`** показує **оперативну пам'ять**: `total` — усього (31 ГБ),
  `used` — зайнято, `free` — повністю вільно, `buff/cache` — кеш ядра (звільняється
  за потреби), а **`available`** — реально доступно для нових програм (найважливіша
  колонка). Окремий рядок `Swap` — підкачка на диску.
- Альтернатива «у Мб»: `free -m`; жива картина пам'яті та CPU — у `top`/`htop`
  (рядки `MiB Mem` та `MiB Swap`).
